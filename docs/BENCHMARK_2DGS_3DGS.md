# 3DGS vs 2DGS Performance Benchmark (Fern Dataset)

This document summarizes the performance comparison between 3D Gaussian Splatting (3DGS) and 2D Gaussian Splatting (2DGS) using the `jax-gs` implementation on the Fern dataset.

## Environment
- **Device:** TPU v2/v3 (single core benchmark)
- **Dataset:** Fern (LLFF) @ 1/8 resolution
- **Gaussians:** ~10,000 (initialized from COLMAP SfM points)
- **Resolution:** 504x378

## Results

| Metric | 3DGS | 2DGS | 2DGS Slowdown |
| :--- | :--- | :--- | :--- |
| **Renderer (Latency)** | 259.96 ms | 389.21 ms | **1.50x** |
| **Renderer (FPS)** | 3.85 FPS | 2.57 FPS | - |
| **Training Step (Latency)** | 1287.35 ms | 1390.58 ms | **1.08x** |
| **Training Step (it/s)** | 0.78 it/s | 0.72 it/s | - |

## Analysis

### Rendering
2DGS rendering is **1.5x slower** than 3DGS. This overhead is attributed to:
1.  **More complex projection:** Projecting 2D primitives (splats) involves computing tangent vectors and specialized 2D covariance matrices.
2.  **Auxiliary Outputs:** 2DGS generates additional maps (depth, depth squared, normals, and accumulation weights) required for the specialized 2DGS loss functions.

### Training
Surprisingly, the **training step overhead is only 8%**. This suggests:
1.  **Shared Overhead:** The majority of the training step time is spent on common operations: gradient computation via `jax.grad`, optimizer state updates (Adam), and backpropagation through the shared parts of the rasterizer.
2.  **Efficient Loss Functions:** JAX's XLA compiler effectively optimizes the additional depth distortion and normal consistency loss terms used in 2DGS.

## Multi-Device Performance
In a separate data-parallel training run (across 4 TPU devices), the system reached **~12.5 it/s** for 2DGS, demonstrating excellent scalability.
